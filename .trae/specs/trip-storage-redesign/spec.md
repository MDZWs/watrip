## 1. 功能概述

重构行程存储架构与删除联动机制。当前存在两套并行且行为不一致的存储系统（`AIMemory` 本地回收站 + `TripStorage` 云端硬删），导致"删除行程后社区帖子仍存在"的 P0 bug。本期将统一为"本地为主 + 云端附属"架构，修复删除联动，调整数量限制，并优化多行程管理体验。

**不做项**：旅行册子（专辑）功能推迟到下期迭代。

---

## 2. 用户决策摘要

| 维度 | 决策 |
|------|------|
| 架构方向 | 本地为主存储 + 云端附属（自动同步+发布） |
| 数量限制 | 本地 20 条上限 / 社区发布 ≤10 条 / 云端自动同步无独立限制 |
| 删除策略 | 全部硬删无回收站（移除现有 AIMemory 回收站机制） |
| 删除联动 | 本地删 → 云端删 → 社区帖子删；别人收藏也消失 |
| 下架功能 | 保留：已发布行程可下架（仅自己可见，社区不显示，别人收藏消失） |
| 登录门槛 | 必须登录才能创建行程 |
| 20 条溢出 | 提示用户选择（删除哪个 / 归档哪个 / 取消） |
| 换设备同步 | 智能合并 + 本地保留 + 元数据同步 |
| 离线创建 | 本地存 + `pendingSync` 标记，联网自动上传 |
| 旅行册子 | 暂不做，下期迭代 |

## 4. 验收标准（AC）

### AC-1：本地创建行程（已登录，未达上限）
**Given** 用户已登录，当前本地行程数 < 20
**When** 用户点击"新建行程"或"AI 规划"并保存
**Then**
- 行程保存到 localStorage
- 云端自动同步上传（不弹"是否同步"确认框）
- 列表立即显示新行程
- 上传成功后行程带 `cloudId` 标记

### AC-2：本地创建行程（达到上限）
**Given** 用户已登录，当前本地行程数 = 20
**When** 用户点击"新建行程"
**Then**
- 阻止创建
- 弹出选择面板：「删除一个旧行程」/「归档一个旧行程」/「取消」
- 选择"删除"→进入选择模式，选中后删除再创建
- 选择"归档"→进入选择模式，选中后归档（归档不计入 20 条）再创建
- 选择"取消"→返回

### AC-3：删除未发布行程
**Given** 用户有一个未发布到社区的行程（status=private）
**When** 用户长按删除并确认
**Then**
- 本地 localStorage 删除（硬删，不进回收站）
- 云端 trip_templates 记录删除（硬删）
- 列表立即移除该项
- Toast 提示"已删除"

### AC-4：删除已发布行程（核心修复）
**Given** 用户有一个已发布到社区的行程（status=published），被 5 人收藏、3 人复制
**When** 用户长按删除并确认
**Then**
- 本地 localStorage 删除（硬删）
- 云端 trip_templates 记录删除（硬删，因 ON DELETE CASCADE 会联动删 likes/favorites）
- 社区列表不再显示此行程
- 5 人的收藏列表中此行程消失
- 3 人复制到他们本地的副本保留（复制时已是独立副本）
- Toast 提示"已删除"

### AC-5：下架已发布行程
**Given** 用户有一个已发布到社区的行程
**When** 用户在行程详情页点击"下架"
**Then**
- 云端 trip_templates 的 status 改为 private
- 社区列表不再显示
- 别人收藏列表中此行程消失
- 用户自己"我的行程"中仍可见，标记为"已下架"
- Toast 提示"已下架，仅自己可见"
- 可再次"重新发布"

### AC-6：发布到社区达上限
**Given** 用户已发布 10 条行程到社区
**When** 用户尝试发布第 11 条
**Then**
- 阻止发布
- 提示"社区发布上限 10 条，请先下架或删除已发布行程"
- 引导到"我的发布"列表

### AC-7：未登录访问创建入口
**Given** 用户未登录
**When** 点击"新建行程"或"AI 规划"或"手动规划"
**Then**
- 拦截跳转
- 弹出登录引导弹窗："登录后即可创建行程"
- 提供"去登录"按钮
- 登录成功后继续原操作

### AC-8：换设备登录同步
**Given** 用户在设备 A 有 15 条本地行程，其中 5 条已发布
**When** 用户在设备 B 登录同一账号（设备 B 本地无数据）
**Then**
- 从云端拉取 15 条行程到本地
- 按 `updatedAt` 智能合并去重
- 元数据（status / likes / favorites / copies）同步更新
- 不弹"是否同步"确认框，自动完成
- 同步完成后 Toast 提示"已同步 X 条行程"

### AC-8b：换设备登录同步（本地已有数据）
**Given** 用户在设备 A 有 15 条行程，设备 B 本地有 8 条（其中 3 条与设备 A 重叠）
**When** 用户在设备 B 登录
**Then**
- 设备 B 本地 8 条保留
- 云端 15 条智能合并：重叠的 3 条按 `updatedAt` 取新
- 不重叠的 10 条新增到本地
- 本地多出的 5 条（云端没有的）自动上传到云端
- 最终本地和云端都是 18 条（8 + 15 - 3 重叠 + 5 本地独有上传... 实际去重后总数）

### AC-9：离线创建行程
**Given** 用户已登录但无网络
**When** 创建并保存行程
**Then**
- 仅存本地，标记 `pendingSync: true`
- 列表显示该行程带"待同步"角标
- 网络恢复后自动上传
- 上传成功后 `pendingSync` 改为 false，角标消失

### AC-10：归档行程
**Given** 用户有一个进行中或已完成的行程
**When** 用户在行程卡片上选择"归档"
**Then**
- 行程从主列表移除，进入"已归档"列表
- 归档不计入 20 条上限
- 已归档列表入口在 tripsPage 顶部或底部
- 归档行程可"取消归档"回到主列表

## 5. 数据模型变更（仅说明，不实现）

### localStorage 结构调整
- `trip_history`：保留，但删除 `trip_trash` 回收站机制
- 每条行程新增字段：
  - `cloudId`：云端 ID（已存在）
  - `pendingSync`：boolean，是否待同步
  - `archivedAt`：timestamp，归档时间（null 表示未归档）

### Supabase trip_templates 表
- 无需 schema 变更（status 字段已支持 private/published）
- 删除策略：物理删除（DELETE），依赖 ON DELETE CASCADE 自动清理 likes/favorites/views
